# PMが記録すべきこと — 観察ポイント・KPI・記録ガイド

**対象:** PMとして何を記録すれば役立つか自信がない方向け
**目的:** AI駆動PMシミュレーションの観察・記録標準を定める

---

## 結論から：最小限これだけ記録すれば十分

```
スプリント毎:
1. スプリント完了率（Done件数 / 全件数）
2. ブロッカー発生件数 と 解消までの日数
3. 想定外だった出来事（1〜3文）

メンバー観察（週次）:
4. 遅れているメンバーと理由
5. PMが取ったアクションとその結果
```

これだけで3スプリント後に「AIのアドバイスが当たっているか」を検証できる。

---

## 記録カテゴリ一覧

### カテゴリ1: スプリントKPI（定量）

| KPI | 計算方法 | 目標値 | 記録頻度 |
|-----|---------|--------|---------|
| スプリント完了率 | Done件数 ÷ 全チケット件数 × 100 | 70%以上 | スプリント終了時 |
| バグ脱出率 | 本番バグ数 ÷ デプロイ件数 × 100 | 15%以下 | スプリント終了時 |
| ブロッカー頻度 | スプリント中のblocked件数 | 2件以下 | スプリント終了時 |
| ブロッカー解消日数 | blockedから解消までの平均日数 | 2日以内 | スプリント終了時 |
| 繰り越し件数 | 次スプリントに持ち越したチケット数 | 2件以下 | スプリント終了時 |

**記録方法:** スプリント終了時に `docs/sprint-logs/sprint-N-summary.md` に記録

---

### カテゴリ2: リードタイム・サイクルタイム（定量）

| 指標 | 定義 | 計算 |
|------|------|------|
| リードタイム | チケット作成 → Done | Jiraの「created」→「resolutiondate」 |
| サイクルタイム | In Progress → Done | 「in-progress」ラベル付与 → Done |
| PRサイクルタイム | PR作成 → マージ | GitHub PR オープン → マージ |

**なぜ重要か:** チャーリー（怠惰）はサイクルタイムが長い。ダナ（無能）はリードタイムが長い。どちらが問題かを区別できる。

**記録方法:** Jira の「作成日」「完了日」から手計算 or スプリントレポート生成時に自動集計

---

### カテゴリ3: メンバー別観察（定性 + 定量）

**週次でチームメンバーごとに記録する:**

```markdown
## 週次メンバー観察 YYYY-MM-DD

### チャーリー（dev-a）
- 状況: Sprint Day 7。JZZG-7（ユーザー設定画面）がまだTodo
- 気になる点: 4日間更新なし。過去パターン通り後半に動く可能性
- PMアクション: Slack DM で「何か困っていることはあるか」確認
- アクション結果: 「来週には終わる」と返信。具体的な進捗なし
- 次の観察ポイント: Day 10（残り4日）で進捗確認

### ダナ（dev-a）
- 状況: JZZG-8（エラーハンドリング）をIn Progress中
- 気になる点: PRのコメントに「どこから始めるか迷っている」と書いていた
- PMアクション: アリスにペアレビュー依頼
- アクション結果: アリスが30分サポート。PRドラフト提出
```

**なぜ記録するか:**
- 「問題が予測できた」→ AIリスク検出の精度向上
- 「アクションが効果あり/なし」→ Skills文書の精度向上
- 「同じ問題が繰り返す」→ チームシナリオの更新

---

### カテゴリ4: PMアクションログ（最重要）

**PMが何かアクションを取った時は必ず記録する:**

```markdown
## PMアクションログ

日時: 2026-03-10 (Sprint 1 Day 9)
問題: チャーリーのJZZG-7が無更新5日
アクション: 1on1 MTG設定 → 実はAPIの仕様が不明で止まっていた
結果: イブ（バックエンド）に仕様確認を依頼 → 翌日解決
学び: 「報告しない」タイプのメンバーは定期的な1on1が有効
```

**記録ファイル:** `docs/sprint-logs/pm-actions.md`

---

### カテゴリ5: AI活用ログ（このシミュレーション固有）

**AIを使ったPMアクションの記録:**

```markdown
## AI活用ログ

日時: 2026-03-05
使用Skill: pm-risk-detect
入力: Sprint 1 Day 4 のチケット一覧
AIの出力: 「チャーリーのJZZG-7がdelay_risk。グレースのレビュー待ちが48h超」
PMの判断: グレースへのレビュー催促は今日実施。チャーリーは様子見
実際の結果: グレースは当日夜にレビュー完了。チャーリーは翌日も無更新
評価: AI予測 ✅（グレースリスク）+ ✅（チャーリーリスク）
```

**なぜ記録するか:** Skills文書の精度改善のため。3スプリント後に「AIが正しかった率」を計算できる。

## 記録のしすぎを防ぐ原則

1. **「決断した理由」を記録する（経緯ではなく）**
   - ❌ 「チャーリーのタスクが遅れていた。3日後に確認した。5日後にMTGした。」
   - ✅ 「3日無更新で1on1設定。理由: 過去パターンで終盤に問題が表面化するため」

2. **「予想と結果の乖離」を記録する**
   - AIが「リスクなし」と言ったのに問題が起きた場合
   - AIが「リスクあり」と言ったのに何も起きなかった場合

3. **週に1回30分、スプリントに1回1時間が目安**
   - 記録に時間をかけすぎると本業の妨げになる

4. **Jiraのコメントを一次情報として活用**
   - チケットのコメントに仮想タイムスタンプ付きで記録すれば二重管理不要

## 会社横展開時の注意事項

このガイドを実際の職場で使う場合:
- `docs/scenarios/team-scenario.yaml` のメンバー特性を実際のチームに合わせて差し替える
- 「怠惰」「無能」という表現は内部ドキュメントのみ使用し、対外的には「要サポート」等の表現に変更
- KPI目標値はプロジェクト規模・チーム成熟度に合わせて調整
- シミュレーション用のリスクシナリオは削除し、実際のプロジェクトリスクに置き換える
